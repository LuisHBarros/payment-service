Fluxo de Depósito com Stripe

Depósito via Stripe Checkout — o backend gera a sessão e retorna checkoutUrl. O pagamento é confirmado assincronamente via webhook, processado com Outbox Pattern.

💳
8
etapas
🔗
Webhook
async confirm
🔑
Sig
HMAC verify
1
HTTP · Client
POST /api/deposits/checkout
Cliente envia walletId e amount. JWT necessário. Resposta: { checkoutUrl }.
DepositRequest Bearer Token
2
Controller · Strategy Pattern
DepositController
Resolve o provider via Map<String, PaymentProvider> injetado. Seleciona "stripe" e delega a criação da sessão.
Strategy Pattern PaymentProvider Map injection
⬡ Stripe — External Payment Provider
3
Stripe · CheckoutSession
StripeCheckoutStrategy
Cria Stripe Checkout Session via SDK. Retorna checkoutUrl — frontend redireciona o usuário para o Stripe.
Stripe SDK checkoutUrl Redirect
usuário paga no Stripe
4
Stripe Webhook · Async
POST /webhooks/stripe
Stripe envia checkout.session.completed com Stripe-Signature header. Disparo assíncrono após confirmação do pagamento.
checkout.session.completed Stripe-Signature
5
Security · Signature Verification
StripeWebhookHandler
Verifica assinatura HMAC-SHA256 com Stripe.constructEvent(). Extrai walletId e amount do metadata.
HMAC-SHA256 Signature Verify 400 se inválida
6
UseCase · Application Layer
CompleteDepositUseCase
Idempotência via stripeSessionId. Credita a wallet e persiste DepositCompletedEvent no Outbox — @Transactional.
Idempotent @Transactional Outbox Pattern
@Scheduled poll
7
Kafka Producer · Outbox
OutboxPublisher → Kafka
@Scheduled publica DepositCompletedEvent no tópico Kafka para consumers downstream (notificações, relatórios).
deposits.completed at-least-once
✓
Response · Completed
Depósito Concluído 🎉
Webhook responde 200 para Stripe. Frontend faz polling em GET /api/deposits/{id} e exibe confetti ao detectar COMPLETED.
200 OK Polling COMPLETED
100%

Fluxo de Autenticação JWT

Autenticação stateless com JWT — registro, login, geração de token e invalidação via Redis. Rate limiting protege os endpoints de auth contra brute force.

🔐
7
etapas
⏱
Redis
token blacklist
🛡
BCrypt
password hash
1
HTTP · Public Endpoint
POST /api/auth/login
Endpoint público com rate limiting via Redis (5 req/min por IP). Aceita { email, password }.
Rate Limited LoginRequest Public
2
Controller · Application
AuthController
Valida request com @Valid e delega para AuthService. Retorna { token, expiresIn }.
@Valid @RestController
3
Repository · PostgreSQL
UserRepository.findByEmail()
Busca usuário por email. Se não existe, retorna 401 — nunca diferencia "email inválido" de "senha errada".
JPA Optional<User> 401 se não encontrado
4
Security · Password Verify
BCryptPasswordEncoder
Compara hash BCrypt com password fornecido. Timing-safe. Retorna 401 se não corresponder.
BCrypt Timing-safe 401 se inválido
5
Security · JWT Generation
JwtTokenProvider.generate()
Gera JWT assinado com HS256. Claims: sub (userId), roles, iat, exp. TTL configurável.
JJWT HS256 sub / roles / exp
200 OK
✓
Response · Token
{ token, expiresIn }
Client recebe JWT e usa em Authorization: Bearer. Para logout, token é adicionado à blacklist no Redis com TTL igual ao exp do token.
200 OK Bearer Token Redis Blacklist
100%
